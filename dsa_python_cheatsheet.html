Union-Find (DSU)

parent=list(range(n))
rank=[0]*n
def find(x):
  if parent[x]!=x:
    parent[x]=find(parent[x])
  return parent[x]
def union(x,y):
  px,py=find(x),find(y)
  if px==py: return False
  if rank[px]<rank[py]: px,py=py,px
  parent[py]=px
  if rank[px]==rank[py]:
    rank[px]+=1
  return True

Dijkstra Template

import heapq
def dijkstra(graph, src, n):
  dist=[float('inf')]*n; dist[src]=0
  pq=[(0,src)]
  while pq:
    d,u=heapq.heappop(pq)
    if d>dist[u]: continue
    for v,w in graph[u]:
      if dist[u]+w<dist[v]:
        dist[v]=dist[u]+w; heapq.heappush(pq,(dist[v],v))
  return dist

Union-Find Template

class DSU:
  def __init__(self,n):
    self.p=list(range(n)); self.rank=[0]*n
  def find(self,x):
    if self.p[x]!=x: self.p[x]=self.find(self.p[x])
    return self.p[x]
  def union(self,x,y):
    px,py=self.find(x),self.find(y)
    if px==py: return False
    if self.rank[px]<self.rank[py]: px,py=py,px
    self.p[py]=px
    if self.rank[px]==self.rank[py]: self.rank[px]+=1
    return True

⏱ TIME + SPACE COMPLEXITY — EXPLAINED
What is Big O?

Big O describes how your algorithm's time (or memory) grows as input size n grows. It's the worst-case upper bound — how bad can it get?

The 3 Notations
NotationMeaningUse
O(f(n))Upper bound — worst caseMost common in interviews
Ω(f(n))Lower bound — best caseRarely used in interviews
Θ(f(n))Tight bound — exact growthWhen best=worst case
Intuition for Each Complexity
O(·)IntuitionExample
O(1)Same time regardless of input sizeArray access a[5], dict lookup
O(log n)Halves the problem each stepBinary search — 1M items needs ~20 steps
O(n)Touch every element onceLinear scan, one for loop
O(n log n)Sort + linear = n times log n workMerge sort, heapsort
O(n²)For every element, touch every other elementNested loops, bubble sort
O(2ⁿ)Every choice doubles the workSubsets, brute force backtracking
O(n!)Try every orderingPermutations, brute force TSP
💡 Interview Rule of Thumb: n=10⁶ → need O(n) or O(n log n). n=10³ → O(n²) ok. n=20 → O(2ⁿ) ok.
❌ Common mistake: Counting O(n) + O(n) = O(n), not O(2n). Constants and lower terms get dropped.
💡 Amortized complexity: Dynamic array append is O(1) amortized — occasional O(n) resizes average out over many operations.
📋 ARRAYS — PRACTICAL GUIDE
Why is Arrays the most important topic?

70% of LeetCode problems use arrays. All the key patterns — prefix sum, sliding window, two pointers, binary search — are array techniques. Master these and you solve most interview problems.

Pattern 1: Prefix Sum — "precompute range queries"
When? Any problem asking for "sum of subarray from i to j". Instead of summing each time in O(n), precompute cumulative sums so any range query is O(1).
# Example: subarray sum equals k
# Naive: O(n²) — check every pair
# Smart: prefix sum + hash map O(n)
nums = [1, 2, 3, -2, 5]; k = 5
prefix = {0: 1}; cur = res = 0
for x in nums:
    cur += x
    if cur - k in prefix: res += prefix[cur-k]
    prefix[cur] = prefix.get(cur, 0) + 1
Pattern 2: Sliding Window — "variable-size subarray"
When? "Longest/shortest subarray with property X". Grow window right, shrink from left when invalid. Avoids nested loops → O(n) instead of O(n²).
❌ Mistake: Resetting window instead of shrinking. When window becomes invalid, shrink it from left — don't start over.
Pattern 3: Two Pointers — "sorted array pair finding"
When? Sorted array + find pair/triple with target sum. Start at both ends: too small → move left pointer right. Too big → move right pointer left.
Pattern 4: Kadane's Algorithm — "maximum subarray"

The insight: at each element, decide — is it better to start fresh from here, or extend the previous subarray? cur = max(x, cur + x)

💡 If asked "subarray" or "contiguous" → think sliding window or prefix sum first.
🌲 TREES — INTUITION + STRATEGY
How to think about Trees

Most tree problems follow the same insight: a tree is a recursive structure. Every node is the root of its own subtree. So "solve for the whole tree" = "solve for root given results from left and right subtrees."

DFS vs BFS — When to use each
Use DFS whenUse BFS when
Need to explore full paths (root-to-leaf)Need level-by-level processing
Need max depth, height, diameterNeed shortest path to node
Need to compare left vs right subtreeNeed rightmost node at each level
Checking properties recursivelyFinding minimum depth
The "Return Up" Pattern — Most Important Tree Technique
Most hard tree problems: compute something at each node using results from its children, and pass a result UP to the parent. Example: diameter — each node passes its max depth up. The diameter is max(left_depth + right_depth) tracked globally.
# Pattern: track global result + return local result
self.ans = 0  # global (what we want)
def dfs(node):
    if not node: return 0
    left = dfs(node.left)
    right = dfs(node.right)
    self.ans = max(self.ans, left + right)  # update global
    return 1 + max(left, right)            # return local (depth)
BST Property — The Hidden Superpower
Inorder traversal of BST = sorted array. Use this constantly: validate BST (keep track of min/max bounds), find kth smallest (inorder + count), find range of values, check if two BSTs are identical.
❌ Common mistake on BST validation: checking only that node.left.val < node.val. Wrong! All values in left subtree must be less. Pass min/max bounds through recursion.
💡 Trie is just a tree where each node represents a character. Use it for: autocomplete, word search, prefix matching. Build = O(total chars), Search = O(word length).
Segment Tree — When and Why

Need: (1) range queries (sum/min/max of arr[i..j]) AND (2) point or range updates. Without segment tree: O(n) per query. With it: O(log n) per query and update. Use when constraints say n=10⁵ with 10⁵ queries.

How to approach any graph problem

Step 1: Identify graph type (directed/undirected, weighted/unweighted, DAG?). Step 2: Choose algorithm based on what you need. Step 3: Build adjacency list. Step 4: Handle visited correctly.

Union-Find — The Pattern Recognizer
Union-Find (DSU) answers one question fast: "Are X and Y in the same connected component?" Use it for: number of islands (or connected components), detecting cycles in undirected graph, Kruskal's MST. Path compression + union by rank = O(α(n)) ≈ near O(1).
Topological Sort — Prereqs and Dependencies

When problem says "A must come before B" → topological sort. Key: if after Kahn's algorithm (BFS), the order list has fewer than n nodes → there's a cycle (impossible to complete all tasks).

❌ Dijkstra doesn't work with negative weights — use Bellman-Ford. Dijkstra's greedy assumption breaks with negatives.
💡 Grid problems (islands, paths in matrix) are implicit graphs. Cells are nodes, adjacent cells are edges. Apply BFS/DFS directly with (row, col) as node IDs.
⚡ DYNAMIC PROGRAMMING — FROM ZERO TO HERO
What is DP really?

DP = recursion + memoization. Any recursive solution where you recompute the same subproblem multiple times can be optimized with DP. The key insight: store results of already-solved subproblems.

4-Step DP Framework
StepQuestion to AskExample (Climb Stairs)
1. Define stateWhat does dp[i] represent?dp[i] = ways to reach step i
2. Base caseSmallest subproblem answer?dp[0]=1, dp[1]=1
3. TransitionHow does dp[i] depend on previous?dp[i] = dp[i-1] + dp[i-2]
4. AnswerWhich dp value is the final answer?dp[n]
Memoization (Top-Down) vs Tabulation (Bottom-Up)
Both have same complexity. Use memoization when: easier to think recursively, not all subproblems needed. Use tabulation when: all subproblems needed, want to optimize space (rolling array trick).
# Same problem — two approaches (Coin Change)
# Memoization (top-down)
from functools import lru_cache
@lru_cache(None)
def dp(amount):
    if amount == 0: return 0
    if amount < 0: return float('inf')
    return 1 + min(dp(amount - c) for c in coins)

# Tabulation (bottom-up) — space-efficient
dp = [float('inf')] * (amount + 1); dp[0] = 0
for i in range(1, amount + 1):
    for c in coins:
        if c <= i: dp[i] = min(dp[i], dp[i-c] + 1)

❌ Mistake: Wrong state definition. If your state doesn't fully capture what you need to make the next decision, you'll get wrong answers. Add more dimensions to the state.
💡 Space optimization: Most 2D DP can use 2 rows (rolling array). Some 1D DP need only 2 variables (like Fibonacci). Reduces O(n²) space to O(n) or O(1).
💡 Bitmask DP: When n ≤ 20, you can represent which elements have been "used" as a bitmask. dp[mask] = best result using the set of elements represented by mask.
↩ BACKTRACKING + BINARY SEARCH + GREEDY
Backtracking — Systematic Exploration

Backtracking is structured brute force. It builds a solution step by step, and backtracks (undoes the last step) when it hits a dead end. Key: every recursive call MUST undo what it did ("choose → explore → unchoose").

3 Classic Backtracking Problems
ProblemKey InsightDeduplication?
SubsetsAt each index: include or skip. Pass i+1 to avoid repeats.Sort + skip duplicates
PermutationsEvery element can go in every position. Use visited[] array.Sort + skip if used[i-1] not used
CombinationsLike subsets but fixed size. Pass i+1 (no reuse) or i (reuse).Sort + skip same value at same depth
💡 Pruning is everything in backtracking. Check constraints BEFORE recursing, not after. "If remaining elements can't fill the target → return immediately."
Binary Search — Beyond Simple Search

Binary search isn't just for finding a value. The real power: "Binary search on the answer." If you can ask "is X a valid answer?" and the validity is monotone (once valid, stays valid), binary search the answer space.

Template thinking: Define what "too small" and "too large" means. Binary search between min and max possible answer. For each midpoint, call feasible(mid) to check if it works.
# Example: "Minimum capacity to ship all packages in D days"
# Binary search on capacity (answer), not index!
def feasible(capacity):
    days = ships = 1
    for w in weights:
        if ships + w > capacity: days += 1; ships = 0
        ships += w
    return days <= D
lo, hi = max(weights), sum(weights)
while lo < hi:
    mid = (lo + hi) // 2
    if feasible(mid): hi = mid
    else: lo = mid + 1
Greedy — When Local Best = Global Best

Greedy works when making the locally optimal choice at each step leads to the globally optimal solution. The challenge: proving your greedy choice is safe. Common pattern: sort by end time (intervals), sort by some ratio, track maximum reach (jump game).

❌ Greedy doesn't always work! If a problem has "dependencies" or choices that affect future choices non-monotonically → use DP instead. When in doubt: try greedy, then check if DP is needed.

  • Off-by-one errors: Range is [l, r] or [l, r)? Is the window r-l or r-l+1 elements? Write examples first.
  • Empty input: Always check if len(nums) == 0 at the start. Heap operations on empty list crash.
  • Integer overflow: Python handles big ints natively, but for C++/Java, use long. Note when sum could overflow.
  • Modifying list while iterating: Never! Create a new list or iterate a copy.
Trees
  • Not handling None nodes: Always check `if not node: return base_value` first in recursive functions.
  • BST validation: Don't check only immediate parent — pass bounds through the recursion.
  • Diameter vs depth confusion: Diameter passes through a node (left_depth + right_depth). Return max(left, right) + 1 for depth.
Graphs
  • Forgetting to mark visited: For BFS/DFS, mark visited WHEN you add to queue, not when you pop. Otherwise you visit nodes multiple times.
  • Dijkstra with negative weights: Just doesn't work. Use Bellman-Ford.
  • Cycle in topological sort: If topo_order length < n, there's a cycle — the tasks can't all be completed.
DP
  • Wrong base case: dp[0] = ? — think carefully. Off-by-one in indexing ruins everything.
  • Wrong transition direction: In 0/1 knapsack, iterate weight backwards to prevent reusing same item.
  • Forgetting to handle -∞ / +∞: Initialize with float('inf') or float('-inf') appropriately.
Binary Search
  • Infinite loop: Make sure l and r converge. Use l<r not l<=r for "find leftmost" pattern.
  • Wrong boundary: l=0, r=n (exclusive) or r=n-1 (inclusive)? Choose one style and be consistent.
❌ Biggest interview killer: Not testing your code! Always trace through your solution with the example before declaring done.

LeetCode-Specific Tips
  • Read constraints first: n ≤ 20 → backtracking ok. n ≤ 10⁵ → need O(n log n). n ≤ 10⁶ → need O(n). Constraints tell you the expected algorithm!
  • Python is slow for large n: If TLE, consider: use C++ I/O tricks (sys.stdin), reduce function calls, use built-ins (max, sum, sorted) instead of manual loops.
  • Heap for top-K problems: Always use a heap of size K (min-heap for K largest). Never sort the whole array if you only need top K.
  • Hash maps everywhere: When you need O(1) lookup for "have I seen X before?" → hash set. "How many times have I seen X?" → Counter or defaultdict(int).

🏔 HEAPS + PRIORITY QUEUES — EXPLAINED
What is a Heap?

A heap is a complete binary tree stored as an array where every parent is smaller than its children (min-heap) or larger (max-heap). The root is always the min (or max). This guarantees: peek min/max in O(1), insert/delete in O(log n).

The #1 Heap Pattern: K-sized Heap
The key insight: maintain a heap of exactly K elements. To find K largest → use a MIN heap of size K. Why? The min of the K largest is always at the top. When a new element is bigger than the smallest in your K-group, swap it in.
Why Python only has min-heap

Python's heapq is always a min-heap. To simulate a max-heap, negate all values: push -x instead of x, and negate the result when you pop: -heapq.heappop(h). For tuples: (-priority, item).

Kth Largest ElementMin-heap of size K. Pop when size > K. Answer = heap[0]O(n log k)
Merge K Sorted ListsPush first node of each list. Pop min, push its next.O(n log k)

💡 heapq.nlargest(k, nums) and heapq.nsmallest(k, nums) are built-in. Use them for one-off queries. For streaming data, maintain a heap manually.
❌ Common mistake: using sorted() to find Kth largest — that's O(n log n). A heap gives O(n log k) which is much better when k << n.
🔤 STRINGS — DEEP INTUITION
Strings = Arrays of Characters

Most string problems are array problems in disguise. The same patterns apply: sliding window for substrings, two pointers for palindromes, hash maps for frequency tracking. The string-specific tools are: KMP for pattern matching, Trie for prefix queries, Rolling Hash for fast comparison.

Sliding Window for Strings — The Universal Template
For "longest/shortest substring with condition X": expand window to the right, shrink from the left when condition is violated. Track frequency of characters in the window with a Counter/dict.
# Template: longest substring with at most k distinct chars
count = {}; l = res = 0
for r, c in enumerate(s):
    count[c] = count.get(c, 0) + 1
    while len(count) > k:          # window invalid
        count[s[l]] -= 1
        if count[s[l]] == 0: del count[s[l]]
        l += 1
    res = max(res, r - l + 1)
KMP — Why Bother?

Naïve string matching: for each position in text, check if pattern matches = O(n*m). KMP precomputes a "failure function" that tells you, when a mismatch happens, the longest proper prefix of the pattern that is also a suffix. This lets you skip redundant comparisons → O(n+m).

💡 Rolling hash (Rabin-Karp): hash a window, slide it O(1) per step. Perfect for finding duplicate substrings of length k in O(n).

Why Bit Manipulation?

Bit tricks replace slow operations with O(1) bit-level operations. They appear in ~10% of LeetCode problems. Most important: the XOR trick for finding unique elements.

Bitmask DP — When n ≤ 20

When you need to track which elements from a set of n have been "used", use a bitmask. An integer with n bits can represent any subset. Iterate through all 2ⁿ subsets in O(2ⁿ) instead of O(n!) for brute-force permutations.

💡 Subset enumeration: to enumerate all subsets of a bitmask, use: sub = mask; while sub: process(sub); sub = (sub-1) & mask. This visits each subset exactly once in O(3ⁿ) total.
📚 MONOTONIC STACK + QUEUE — EXPLAINED
What is a Monotonic Stack?

A monotonic stack maintains elements in increasing or decreasing order. Whenever a new element violates the order, you pop until the order is restored. This lets you efficiently find the "next greater/smaller element" for every position in O(n) instead of O(n²).

The Core Insight
When you pop element X because new element Y is larger (in an increasing stack), it means Y is the "next greater element" for X. This is why monotonic stacks solve NGE problems: each element is pushed and popped at most once → O(n) total.
Monotonic Increasing vs Decreasing
TypePop conditionFindsUse case
Increasing (bottom→top)Pop when new < topNext smaller elementStock span, trapping water
Decreasing (bottom→top)Pop when new > topNext greater elementNGE, daily temperatures
Largest Rectangle in Histogram — Classic Hard Problem

Key insight: for each bar, find the nearest shorter bar on its left and right — that's the width of the rectangle with that bar as the shortest. A monotonic increasing stack does this implicitly: when you pop a bar, the current bar is its right boundary, and the new stack top is its left boundary.

Monotonic Deque — Sliding Window Maximum
Same idea, but with a deque so you can remove from both ends. Maintain decreasing order. Remove from front when index is out of window. Remove from back when new element is larger (they can never be the maximum while the larger element is in the window).
💡 Pattern trigger: "Next greater/smaller", "stock span", "histogram", "largest rectangle", "sliding window maximum" → Monotonic stack/deque.
❌ Mistake: storing values instead of indices. Always store INDICES so you can compute distances/widths. Look up values via a[index].
🔬 ADVANCED DSA — WHEN AND WHY
Segment Tree vs Fenwick Tree vs Sparse Table

All three support fast range queries. Choose based on whether you need updates and what type of query you need.

StructureBuildQueryUpdateUse when
Prefix Sum ArrayO(n)O(1)O(n)Static array, sum queries only
Fenwick Tree (BIT)O(n)O(log n)O(log n)Prefix sum with point updates
Segment TreeO(n)O(log n)O(log n)Any range query + point/range updates

Fenwick Tree Intuition

Each index i in a Fenwick tree "is responsible for" a range of elements determined by the lowest set bit of i. Updating index i propagates forward by adding the lowest set bit: i += i & (-i). Querying sums backward by removing the lowest set bit: i -= i & (-i).

Suffix Array — When to Use

A suffix array lists all suffixes of a string in sorted order. Combined with the LCP (Longest Common Prefix) array, it solves: longest repeated substring, number of distinct substrings, longest common substring of two strings — all in O(n log n) or O(n) space vs O(n²) with brute force.

💡 In competitive programming, if you see "range queries on array with updates" and n = 10⁵, immediately think Segment Tree or Fenwick Tree.
💡 Python: use the sortedcontainers library's SortedList for O(log n) insert/delete/index in a sorted list — replaces balanced BST in most contest scenarios.

When You're Completely Stuck (Interview Protocol)
  1. Re-read the problem. Did you miss a constraint? "Sorted array" means binary search or two pointers.
  2. Try a smaller example. Work manually with n=3 or n=4. The pattern becomes visible.
  3. State what you know. "I know the brute force is O(n²) nested loops. The bottleneck is the inner loop — can I replace it with a hash map or binary search?"

  5. Ask for a hint. Interviewers expect this. "Would it help if I considered sorting first?" is a great way to get a nudge.
